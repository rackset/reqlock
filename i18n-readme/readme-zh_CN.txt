WordPress 出站防火墙：一键拦截外部及缓慢的第三方请求，兼顾韧性、性能、隐私与离线开发。

== Description ==

**ReqLock** —— 也写作 **RequestLock** 或 **Request Lock** —— 是一款面向 WordPress 的出站
（egress）防火墙。它掌控你的站点*对外*发往互联网的每一次调用，覆盖请求的两端：**服务器端**
（PHP / WP HTTP API）和**浏览器端**（页面渲染出的 HTML）。一个主开关，让你的站点完全掌控自己的出站流量。

如今的 WordPress 站点很“吵”：更新检查、授权回拨、统计分析、标签管理器、外部字体、嵌入式小工具、
AI 接口，以及各种“phone-home”调用，都会去访问你无法掌控的服务器。当这些服务器缓慢、被封锁或宕机时，
你的页面和后台都会受拖累 —— 而它们每一个都是访客数据外泄的出口。ReqLock 让你随时即刻、可逆地切断
这些流量，无需编辑主题文件，也无需逐一排查插件。

**一个开关，四种用途：**

* **韧性** —— 当外部互联网被**切断或受限**（中断、区域性断网、上游故障）时，让站点保持可用。页面从本地资源
  提供，wp-admin 不再卡在无响应的请求上。
* **性能** —— 缓慢或失效的第三方调用**立即失败**，而不是让前台和后台页面在漫长的超时上空等。
* **隐私** —— 剥离统计分析、追踪器、外部字体和 phone-home 请求，让访客的任何信息都不会离开服务器。
* **开发** —— 把任意安装变成自给自足、**可离线运行**的环境：没有外部调用、不从测试副本发送追踪数据、
  开发时也无需等待远程接口。

= 使用场景 =

* **断网 / 封锁下的韧性。** 当上游连接被限流或封锁时，普通的 WordPress 站点会在每一次外部调用上卡住。
  打开 ReqLock，站点便继续从本地资源提供服务 —— 后台同样如此。
* **为迟缓的站点提速。** 单个缓慢的统计或字体主机就可能给每个页面增加数秒加载时间。ReqLock 让这些调用
  快速失败，而不是阻塞渲染。
* **隐私 / 零追踪部署。** 运行一个可被证明不发出任何第三方请求的站点 —— 适用于隐私优先的项目、内部工具，
  以及对合规敏感的环境。
* **本地与预发布开发。** 把生产环境克隆到笔记本或预发布服务器上，ReqLock 会阻止它“phone-home”：测试副本
  不会触发统计、不会发出授权回拨、也不会在你构建时让 WordPress.org 更新检查拖慢 `wp-admin`。即使拔掉网线，
  站点表现也一致 —— 非常适合离线编码、演示和隔离环境。
* **审计站点对外通信。** “检测到的主机”面板会记录站点访问的每一个外部主机，让你看清主题和插件究竟在联系谁，
  然后决定放行什么、切断什么。

= 拦截哪些内容 =

**服务器端（PHP / WP HTTP API）**

* 发往外部主机的 `wp_remote_*` 出站请求：WordPress.org 更新/版本检查、统计分析、AI 接口（OpenAI、Gemini）、
  远程字体、授权/phone-home 回拨等。它们会立即失败，而不是等待超时。

**浏览器端（渲染出的 HTML）**

* 外部 `<script src>` 和外部 `<link rel="stylesheet">`（例如 Google Fonts）
* 资源提示：`preconnect` / `dns-prefetch` / `preload` / `prefetch`
* 外部 `<iframe>`（替换为干净的本地占位符）
* 内联统计代码片段：Google Analytics / Tag Manager、Microsoft Clarity、Ahrefs、Meta Pixel、Hotjar、Yandex、
  TikTok、Pinterest、LinkedIn、X、Snap、Segment、Plausible
* 可选：外部 `<img>`（透明占位符）

= 关键行为 =

* 你自己的域名及其全部子域名**始终被允许**。
* **允许列表**，用于放行任何其他需要保持可达的主机。
* **检测到的主机面板**会记录见到的每一个外部主机，便于你快速构建允许列表。
* **分类开关** —— 每一层拦截都可独立开启/关闭。
* **wp-config 冲突控制** —— 检测硬编码的 `WP_HTTP_BLOCK_EXTERNAL` 常量，并让你一键禁用（注释掉）或
  重新启用（恢复），从而让 ReqLock 成为外部拦截的唯一开关。所有改动均可逆、带完整性校验且为原子操作。
* **关闭即无效** —— 主开关关闭时，插件什么都不做，因此可以安心保留安装，仅在需要时开启。
* **兼容整页缓存** —— 输出过滤器作为最外层缓冲运行，因此也能覆盖被缓存的页面访问。

== Installation ==

1. 将 `reqlock` 文件夹上传到 `/wp-content/plugins/`，或在**插件 → 安装插件 → 上传插件**处安装 ZIP 包。
2. 在**插件**页面启用 **ReqLock**。
3. 进入**设置 → ReqLock**。
4. 当你想拦截外部请求时（断网期间，或为切断缓慢/追踪类调用）打开**主开关**。它默认**关闭**，因此仅启用插件本身不会带来任何改变。

== Frequently Asked Questions ==

= 启用它会弄坏我的站点吗？ =
不会。主开关关闭时，插件完全不起作用。即使开启，你自己的域名及其子域名也始终被允许。

= 我应该在什么时候打开主开关？ =
任何你想把站点与外部服务隔离的时候：互联网中断/受限期间、为防止缓慢的第三方调用拖慢加载、为隐私剥离追踪器，或为在开发时把预发布/本地副本完全离线。

= 我可以用它来离线开发吗？ =
可以 —— 这是核心用途之一。打开主开关，该安装便不再访问 WordPress.org、统计分析、授权服务器、字体及其他远程主机。你的本地或预发布站点随后在拔掉网线时也能以相同方式加载和运行，绝不从非生产副本发出追踪或 phone-home 调用。

= 它能和缓存插件一起工作吗？ =
可以。输出过滤器作为最外层输出缓冲运行，因此也会过滤被缓存的页面访问（已通过整页缓存插件测试）。

= 它会影响 wp-admin 吗？ =
服务器端的请求拦截在所有位置生效（这会在断网时让后台更快）。浏览器端的 HTML 净化默认仅在前台运行；你也可以选择为 wp-admin 一并启用。

= 我如何在拦截其余内容的同时保留某个外部服务？ =
把它的主机加入允许列表。“检测到的主机”面板会列出 ReqLock 见到的一切，你可以从那里直接复制。

= 那些绕过 WordPress 的自定义 PHP 脚本怎么办？ =
在 WordPress 之外使用原生 `curl`/`file_get_contents` 的独立脚本无法被插件拦截，必须自行处理其外部调用。

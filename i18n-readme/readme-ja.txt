WordPress の送信ファイアウォール。外部や遅いサードパーティ要求をワンスイッチで遮断し、回復力・性能・プライバシー・オフライン開発を実現します。

== Description ==

**ReqLock**（**RequestLock** / **Request Lock** とも表記）は、WordPress 向けの送信（egress）
ファイアウォールです。サイトがインターネットへ*外向き*に行うすべての呼び出しを、リクエストの両側
——**サーバー側**（PHP / WP HTTP API）と**ブラウザー側**（ページが描画する HTML）——で制御します。
1 つのマスタースイッチで、サイトは自らの送信トラフィックを完全に掌握できます。

現代の WordPress サイトは「おしゃべり」です。更新チェック、ライセンス確認、アナリティクス、タグ
マネージャー、外部フォント、埋め込みウィジェット、AI API、各種「phone-home」呼び出しが、あなたの
管理外のサーバーへアクセスします。それらが遅い・ブロックされている・停止していると、ページと
ダッシュボードがその代償を払い、しかもそのどれもが訪問者データの流出口になります。ReqLock を使えば、
テーマファイルを編集することも、プラグインを探し回ることもなく、こうしたトラフィックを即座かつ
可逆的に、必要なときだけ遮断できます。

**1 つのスイッチ、4 つの用途：**

* **回復力** — 外部インターネットが**遮断・制限**されたとき（障害、地域的なシャットダウン、上流の
  不具合）でもサイトを稼働させ続けます。ページはローカル資産から配信され、wp-admin が応答のない
  リクエストでハングしなくなります。
* **性能** — 遅い・応答のないサードパーティ呼び出しは、長いタイムアウトでフロントエンドや管理画面の
  読み込みを止める代わりに、**即座に失敗**します。
* **プライバシー** — アナリティクス、トラッカー、外部フォント、phone-home 要求を取り除き、訪問者に
  関する情報がサーバーから出ていかないようにします。
* **開発** — どんなインストールも、自己完結した**オフライン対応**環境に変えます。外部呼び出しなし、
  ステージングのコピーからのトラッキングなし、作業中にリモート API を待つこともありません。

= ユースケース =

* **障害・遮断への回復力。** 上流の接続が絞られたりブロックされたりすると、通常の WordPress サイトは
  外部呼び出しのたびに止まります。ReqLock を有効にすれば、サイトは管理画面も含めローカル資産から
  配信され続けます。
* **遅いサイトの高速化。** たった 1 つの遅いアナリティクスやフォントのホストが、各ページに数秒を
  追加することがあります。ReqLock はそうした呼び出しを描画を止めずに素早く失敗させます。
* **プライバシー重視・トラッキングなしの公開。** サードパーティ要求を一切行わないことを証明できる
  サイトを運用できます。プライバシー優先のプロジェクト、社内ツール、コンプライアンスに敏感な構成に
  最適です。
* **ローカル・ステージング開発。** 本番をノート PC やステージング環境に複製しても、ReqLock が
  phone-home を防ぎます。テストコピーからアナリティクスが発火せず、ライセンス確認もなく、構築中に
  WordPress.org の更新チェックが `wp-admin` を遅くすることもありません。ネットワークを抜いても
  サイトの挙動は同じで、オフラインでのコーディング、デモ、隔離環境に理想的です。
* **サイトの通信先の監査。** 検出ホストパネルがサイトの到達する外部ホストをすべて記録するので、
  テーマやプラグインが実際にどこへ接続しているかを正確に把握し、何を許可し何を遮断するか判断できます。

= 遮断する対象 =

**サーバー側（PHP / WP HTTP API）**

* 外部ホストへの送信 `wp_remote_*` 要求：WordPress.org の更新／バージョン確認、アナリティクス、
  AI API（OpenAI、Gemini）、リモートフォント、ライセンス／phone-home の確認など。タイムアウトを
  待たずに即座に失敗します。

**ブラウザー側（描画された HTML）**

* 外部 `<script src>` および外部 `<link rel="stylesheet">`（例：Google Fonts）
* リソースヒント：`preconnect` / `dns-prefetch` / `preload` / `prefetch`
* 外部 `<iframe>`（ローカルのプレースホルダーに置換）
* インラインのアナリティクススニペット：Google Analytics / Tag Manager、Microsoft Clarity、Ahrefs、
  Meta Pixel、Hotjar、Yandex、TikTok、Pinterest、LinkedIn、X、Snap、Segment、Plausible
* 任意：外部 `<img>`（透明なプレースホルダー）

= 主な動作 =

* 自分のドメインとそのすべてのサブドメインは**常に許可**されます。
* 到達可能なまま残したい他のホストのための**許可リスト**。
* **検出ホストパネル**が見つかった外部ホストをすべて記録し、許可リストをすばやく構築できます。
* **カテゴリー別スイッチ** — 各遮断レイヤーを個別にオン／オフできます。
* **wp-config の競合制御** — ハードコードされた `WP_HTTP_BLOCK_EXTERNAL` 定数を検出し、ワンクリックで
  無効化（コメントアウト）または再有効化（復元）できます。これにより ReqLock が外部遮断の唯一の
  スイッチになります。変更は可逆的で、整合性チェック付き、かつアトミックです。
* **オフ時は不活性** — マスタースイッチがオフのときプラグインは何もしないため、インストールしたまま
  必要なときだけオンにできます。
* **全ページキャッシュ対応** — 出力フィルターは最も外側のバッファーとして動作するため、キャッシュ
  された表示もカバーします。

== Installation ==

1. `reqlock` フォルダーを `/wp-content/plugins/` にアップロードするか、**プラグイン → 新規追加 → プラグインのアップロード**から ZIP をインストールします。
2. **プラグイン**画面で **ReqLock** を有効化します。
3. **設定 → ReqLock** を開きます。
4. 外部要求を遮断したいとき（障害時、遅い／トラッキング呼び出しを断ち切るとき、またはステージング／ローカルのコピーをオフラインにするとき）に**マスタースイッチをオン**にします。既定は**オフ**なので、有効化しただけでは何も変わりません。

== Frequently Asked Questions ==

= 有効化するとサイトが壊れますか？ =
いいえ。マスタースイッチがオフのとき、プラグインは完全に不活性です。オンのときでも、自分のドメインとそのサブドメインは常に許可されます。

= マスタースイッチはいつオンにすべきですか？ =
サイトを外部サービスから切り離したいときはいつでも：インターネットの障害／制限時、遅いサードパーティ呼び出しによる読み込みの遅延を防ぐため、プライバシーのためにトラッカーを取り除くため、または開発中にステージング／ローカルのコピーを完全にオフラインにするためです。

= オフラインで開発するのに使えますか？ =
はい、これは中心的なユースケースです。マスタースイッチをオンにすると、そのインストールは WordPress.org、アナリティクス、ライセンスサーバー、フォント、その他のリモートホストへの接続をやめます。ローカルやステージングのサイトはネットワークを抜いても同じように読み込まれ・動作し、本番以外のコピーからトラッキングや phone-home を発火することは一切ありません。

= キャッシュ系プラグインと併用できますか？ =
はい。出力フィルターは最も外側の出力バッファーとして動作するため、キャッシュされた表示も対象になります（全ページキャッシュ系プラグインでテスト済み）。

= wp-admin に影響しますか？ =
サーバー側のリクエスト遮断はどこでも適用されます（ネットワーク停止時には管理画面が速くなります）。ブラウザー側の HTML サニタイズは既定でフロントエンドのみ動作し、任意で wp-admin にも有効化できます。

= 一部の外部サービスだけ動かしつつ、ほかを遮断するには？ =
そのホストを許可リストに追加します。検出ホストパネルに ReqLock が見たものがすべて表示されるので、そこからコピーできます。

= WordPress を経由しない独自の PHP スクリプトはどうなりますか？ =
WordPress の外で生の `curl`／`file_get_contents` を使うスタンドアロンのスクリプトはプラグインでは捕捉できないため、外部呼び出しは各自で管理する必要があります。
